# 部署说明

这篇文档记录个人 LLM Agent 部署的设计形态。它不是生产环境运行手册，也不表示这个仓库包含一个完整可部署系统。

## 部署目标

项目关注本地优先的个人 Agent 系统。目标是理解 Agent 工具、模型服务商、本地模型、路由决策和用户确认机制如何在一个实际环境中配合。

部署重点：

- 让敏感工作可以被用户检查。
- 在本地资源和外部模型服务商之间保持清晰边界。
- 为任务选择足够小、足够便宜的模型。
- 记录那些容易隐藏在临时配置里的决策。
- 把安全检查当作部署的一部分，而不是事后补丁。

## 本地优先的基本假设

个人部署可能包含本地模型、本地工具执行、本地文件，以及可选的 hosted API。关键不是所有步骤都必须本地化，而是系统要明确说明哪些步骤在本地完成，哪些步骤会调用外部服务。

每个组件都可以问：

- 这一步是否需要外部网络访问？
- 这一步是否会把私有上下文暴露给 hosted model？
- 用户是否可以检查或中断这个动作？
- 是否有更便宜或更低风险的模型可以完成任务？
- 如果 tool call、model call 或本地进程失败，会发生什么？

## 环境边界

部署边界应该区分：

- 用户文件和 credentials。
- 本地模型运行环境和 cache 目录。
- Hosted model API Key 和模型服务商配置。
- 工具执行权限。
- 日志、trace 和临时产物。

对个人 Agent 系统来说，这些边界能让项目表述更诚实，也能避免把实验描述成无人值守的生产环境系统。

## 运行注意事项

运行姿态应优先选择明确控制，而不是过度自动化：

- 在加入后台自动化前，先从手动执行开始。
- 对会修改文件、账号或外部状态的动作，优先使用 dry run。
- Credentials 不进入文档和源码仓库。
- 实验配置和日常使用配置分开。
- 记录足够排查失败的信息，但避免保存不必要的私人内容。

## 不包含的内容

本仓库不包含可安装的软件包、托管服务或生产环境部署流水线。它记录的是个人 Agent 部署周边的推理、配置和工程实践。
